ACT I · CONCEPT 1 OF 2
AGENTIC DEVELOPMENT

The model stopped suggesting.
It started doing.

Agentic development is software work performed by an AI that runs a loop — not a single completion. It reads the codebase, forms a plan, edits files, runs commands, observes the result, and corrects itself. The human shifts from author to director.

Autocomplete Era

2021–2024 · THE COPILOT
  • Predicts the next token, the next line
  • Human stays in the editor, accepting or rejecting
  • No memory of the task, no plan, no verification
  • Output scoped to what fits on screen

Agentic Era

2025 ONWARD · THE AGENT
  • Plans a multi-step task, then executes it end to end
  • Reads files, runs tests, edits, re-runs — a closed loop
  • Spawns sub-agents, delegates, escalates
  • Output scoped to an entire feature
THE INFLECTION POINT · LATE 2025
Around December 2025, the Agentic Era stopped being a demo and became the default. Developers across the industry began reporting the same thing — that for whole features they were no longer writing a single line of code by hand, only directing the agent that did.

The capability is real. The question this deck answers: what keeps that capability honest?

ACT II · THE DIAGNOSIS
VIBE CODING · DEFINED

Prompt. Accept. Ship.
Repeat until it breaks.

Vibe coding is agentic development with the discipline removed. You describe a vibe, the agent produces code, you skim it, it looks plausible, you accept it. No spec. No acceptance criteria. No verification you didn't write yourself. The feedback loop is vibes, not contracts.

It feels like flow. It is the absence of a process — and absence scales badly.

Why it feels great. Why it fails anyway.

Vibe coding optimizes the first 80% of every task — the part that feels like progress — and silently mortgages the last 20%, the part that determines whether it actually works.

THE COST · A DEFECT GETS MORE EXPENSIVE EVERY HANDOFF

"Looks finished" is the trap. Every handoff multiplies the bill.

Plausible output passes the skim and fails the edge case. The later the defect surfaces, the more it costs — in re-work, in context lost, in trust.

A process that gates at every phase boundary pays for itself on the first prevented incident.

THE MISSING PIECE

There is no contract between
what you meant and what shipped.

In traditional engineering, the spec is the contract. It states the problem, the success criteria, the acceptance tests. Code is judged against it. Vibe coding deletes that document — so there is nothing to judge code against except a vibe in someone's memory.

The fix — coming in ACT III: documents that are contracts, numeric gates that must pass, an archive that traces every idea to its ship, and cascade-aware change.

III.

Spec-Driven Development

The spec is the source of truth. Code is downstream of it.

ACT III · THE PRINCIPLE
SPEC-DRIVEN DEVELOPMENT · THE CORE IDEA

Invert the order.
Specify first. Generate second.

SDD makes one move: the specification — not the code — is the primary artifact. The spec is written, validated, and gated before a line of code exists. Code becomes a derivative of the spec, judged against it, regenerable from it.

Vibe Coding

CODE IS PRIMARY
  • Code is the only artifact that exists
  • Intent lives in chat history, then evaporates
  • "Is it right?" has no reference to check against
  • Change means re-prompt and hope

Spec-Driven Development

SPEC IS PRIMARY
  • The spec is the durable, reviewed artifact
  • Intent is captured, scored, and archived
  • "Is it right?" = "does it satisfy the spec?"
  • Change means update the spec, cascade deliberately
ACT III · THE LINEAGE
WHERE SDD COMES FROM · NOT A FAD, A MATURATION

SDD didn't appear from nowhere. It inherited.

Senior engineers have done versions of this for decades. Spec-Driven Development is the line of practice — written specs, then test-as-spec, then behavior-as-spec — arriving at its agentic form: the spec as the executable source of truth an AI agent builds from.

AgentSpec sits at the right end of that line — spec-as-source, built for an agent to execute.

ANATOMY OF A SPEC · PRD vs THE SDD SPEC

What's inside a spec?

A PRD states what & why for humans. An SDD spec adds the how — it is an executable contract an agent can act on, not just a document people read.

THE CLASSIC · FOR HUMANS
PRD

02Goals & success metrics — measurable
03User stories — how people use it

05Out of scope — explicit boundaries

THE EXECUTABLE CONTRACT · FOR THE AGENT
SDD Spec

AgentSpec's five documents — BRAINSTORM, DEFINE, DESIGN, BUILD_REPORT, SHIPPED — are exactly this SDD spec, split across five gated phases.

DOCUMENTS AS CONTRACTS

Each phase's output is the next phase's input contract.

This is the structural heart of SDD. A phase document is not a deliverable that gets filed away — it is the binding interface the next phase consumes. The DESIGN file's manifest literally becomes the BUILD phase's execution plan. The workflow is deterministic because the contracts are explicit.

No phase guesses. Each one reads a contract and produces the next.

THE 5-PHASE PIPELINE · BRAINSTORM → SHIP

One unified stream. Five phases. One direction.

A raw idea enters at the left. A shipped, archived feature exits at the right. Each phase has its own command, agent, model, input, output, and gate.

GATES, NOT VIBES

Every transition has an explicit, checkable gate.

A phase does not advance because it "feels done." It advances because a specific, named condition is satisfied. The most important gate is numeric — the Clarity Score in Phase 1 — placed exactly where ambiguity is most expensive.

The gate converts a fuzzy judgment — "is this good enough?" — into a computed comparison against a declared threshold.

ACT V · MECHANISM 1 OF 5
THE AGENT TIER SYSTEM

An agent prompt is a conditional schema, not free text.

Every one of the 58 agents is built from one template. The tier: frontmatter field is a switch — it determines which sections of the template are mandatory. A T3 agent missing an Error Recovery section is structurally invalid.

The anti-proliferation gate: before any new agent, verify no existing agent covers >60% — if two overlap >80%, consolidate instead.

ACT V · MECHANISM 2 OF 5
THE CONFIDENCE ENGINE

Confidence is computed — never claimed.

The single most important anti-hallucination rule in AgentSpec: an agent may never self-assess its certainty. It must derive a number from evidence — crossing what the Knowledge Base knows against what MCP servers confirm. That number then decides whether the agent proceeds, asks, or refuses.

STEP 1 · GATHER

Check the KB. Query MCP if the KB is thin. Look for a real codebase example.

STEP 2 · CROSS

Place the evidence on the Agreement Matrix — KB × MCP → a base score.

STEP 3 · ADJUST

Apply modifiers, compare to the impact tier's threshold, act accordingly.

The next two slides show the matrix and the thresholds — the actual machinery.

THE AGREEMENT MATRIX · KB × MCP → CONFIDENCE

A 2×3 truth table that produces a number.

Two knowledge sources, three outcomes each. Where they land decides the base confidence — and the agent's next move.

KB-first by design — "1 KB read + 1 MCP query = 90% coverage." The matrix exists so agents exhaust local knowledge before spending tokens.

MODIFIERS & IMPACT TIERS · THE THRESHOLD

The base score is adjusted — then judged against risk.

Modifiers nudge the matrix score: +0.10 codebase example −0.15 version mismatch −0.05 stale >6mo — then the result meets an impact tier.

Impact TierThresholdBelow ThresholdExamples
CRITICAL0.95REFUSE — explain whySchema migrations, production DDL, deletes
IMPORTANT0.90ASK — get confirmationModel creation, pipeline config, access control
STANDARD0.85PROCEED — with caveatCode generation, documentation, tests
ADVISORY0.75PROCEED — freelyExplanations, comparisons, recommendations

The same template produces a tool that refuses a risky migration and one that explains freely — by changing one number. The risk profile of data engineering is encoded into the decision logic.

ACT V · MECHANISM 4 OF 5
THE KNOWLEDGE BASE · CONFIDENCE-SCORED & FRESHNESS-AWARE

The knowledge base reports its own trustworthiness.

The KB is a database, not a docs folder. Every concept and pattern carries a numeric confidence: score; every domain carries an mcp_validated: date. Knowledge that isn't re-validated decays — automatically.

Self-Reported Confidence

PER FILE
  • Each concept / pattern has a confidence: 0.85–0.95
  • An agent inherits that score when it loads the file
  • The Agreement Matrix adjusts from there — not from zero
  • Knowledge isn't trusted blindly; it's rated

Freshness Decay

PER DOMAIN
  • mcp_validated: a date stamped on every domain
  • Stale > 6 months → a −0.05 confidence modifier fires
  • The KB has a half-life — it ages on purpose
  • shared/anti-patterns.md — anti-knowledge, treated first-class
ACT V · MECHANISM 5 OF 5
WORKFLOW CONTRACTS · THE EXECUTABLE SPEC

One YAML file is the workflow's source of truth.

One file — WORKFLOW_CONTRACTS.yaml — declares how the whole system behaves: every phase, gate, transition and rule lives in it. Every agent and every command reads from this one file, so the rules can't drift, contradict, or hide. Change the file, change the system. It is the single source of truth.
